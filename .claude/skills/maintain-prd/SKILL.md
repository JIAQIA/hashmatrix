# maintain-prd · 数据治理平台 PRD 维护工作流

维护 `prototype/docs/` 下的产品需求文档：**对标华为 DataArts Studio 的功能模块**，用**开源组件实现 + 关键能力自研封装**，按**双前端平面**拆分，全程守**信息红线**。本 skill = 操作手册 + 三块上下文速查（华为参数 / 项目要求 / 现状资源）。

> **事实源**：PRD 的单一事实源是 `prototype/docs/`。本 skill 给"怎么写、对什么"，**详尽明细以 `docs/00-平台总览/` 为准**；二者冲突时**先改文档、再回改本 skill**（定义先行）。

## 0. 动手前必读

维护任何 PRD 前，先读这几处（避免重述导致漂移）：

- `prototype/docs/README.md` —— 总纲：方法论、目录导航、**写作规范与自研定义模板**。
- `prototype/docs/00-平台总览/01-华为DataArts对标矩阵.md` —— 8 组件 × 子能力 × 我方取舍（**华为参数事实源**）。
- `prototype/docs/00-平台总览/02-双平面职责划分.md` —— 使用/管理平面切分规则。
- `prototype/docs/00-平台总览/03-自研概念总账.md` —— 自研概念注册表（SC-1~SC-7）。
- `prototype/docs/00-平台总览/04-一期范围与优先级.md` —— P0/P1/P2、裁剪项。
- 旗舰自研样板：`prototype/docs/06-数据目录/04-数据资产（自研）.md`（自研文档的深度标尺）。
- 架构事实源在主仓 `docs/architecture/`（01 总览 / 03 技术选型 / 05 多租户）。

## A. 华为参数（对标基准）⭐

- **基准产品**：华为云 **DataArts Studio 数据治理中心**。文档入口 `https://support.huaweicloud.com/intl/zh-cn/dataartsstudio/`。
- **对标到"模块 + 子能力"层级**，确保对外能力清单可对标、对内研发不漏项。
- **8 大组件（+ 我方特色）**：

  | # | 组件 | 核心职责 | 我方承载 | 路线 |
  |--|--|--|--|--|
  | 1 | 管理中心 | 数据连接/数据源管理 | data-foundation(Connector SPI)+control-plane | 开源封装 |
  | 2 | 数据集成 | 批量(CDM)+实时(CDC) 20+源迁移 | data-foundation | 开源封装 |
  | 3 | 数据架构 | 主题/标准/模型/指标/审核 | governance | **自研为主** |
  | 4 | 数据开发 | 脚本/作业编排/调度/运维 | platform-common+data-foundation | 纯开源集成 |
  | 5 | 数据质量 | 质量作业/对账/监控/报告 | governance | 开源封装 |
  | 6 | 数据目录 | 元数据/血缘/检索/数据资产 | governance | **自研核心** |
  | 7 | 数据安全 | 分级/敏感识别/脱敏/权限/审计 | security | 开源封装 |
  | 8 | 数据服务 | API 开发/管理/调用 | data-service(横切) | **自研** |
  | + | (Insight·BI) | 报表/大屏 | **—（不建设）** | **不在建设范围**·上层独立 BI 产品（DataEase 类，AD-19） |
  | + | 隐私计算(特色) | MPC/PSI/匿踪 | privacy | 纯开源集成(SecretFlow) |

- **对标 ≠ 照抄，三条裁剪原则**：
  1. **只支持开源与信创数据底座**：裁掉华为云专属库（GaussDB/DWS/MRS/OBS…），换成开源（MySQL/PG/Kafka/Doris/Paimon/MinIO）+ 信创（达梦/金仓/OceanBase）。
  2. **不复刻华为云基础设施依赖**：我们没有云底座（IAM/OBS/VPC），这层须**自建**（control-plane / MinIO / Keycloak），且为私有化刚需。
  3. **对齐到模块与子能力、不对齐交互细节**（见 §E 写作高度）。

> 新模块对标时：先在 `01-华为DataArts对标矩阵.md` 找到对应组件的子能力表，逐项标 ✅保留 / ✂️裁剪 / 🕐暂缓，再落到模块 PRD。

## B. 项目要求（硬约束，违反即否决/违约）

- **★信创是资格门槛**：所有模块须在国产 CPU/OS/云/DB/中间件下可安装运行；信创只作用于"对接的外部数据源"与"最终部署环境"，由部署层 values 切换，**不左右平台自身开源选型**。
- **★刚性工期**：T0 签约 → **2027-02-28 开发完成** → **2027-08-31 终验**；逾期 0.1%/日、超 30 日可解约。→ 自研集中在地基概念，其余尽量开源封装，P2 明确后置。
- **★可用性 99.99%**（年不可用 ≤52.56 分钟）：组件越少越稳；治理下沉 K8s，不引入 Spring Cloud 重治理；自研引擎须无单点、可多副本。
- **★性能硬指标**（由数据基础承接，PRD 不另立）：实时入湖 ≤5s / 70 万条·s；点查 1 万 QPS·TP95≤100ms；实时数仓并发 ≥1000；向量 100T/3 副本；隐私计算千万级求交 ≤100s、亿级匿踪亚秒级；存储 ≥6PB 主备。**承接型模块**（数据集成/数据基础等）可在能力清单/验收要点**引用**这些指标作为可验证项，但**不得新立或加严**指标。
- **★交付物**：全套研制文档 + 源代码（前景知识产权归甲方）+ MBSE 数字模型 → PRD 须可作研制文档输入；**自研概念定义须精准到甲方可接手自维护**。
- **双模交付 + 多租户隔离（C 分层桥接）**：公网 SaaS（我们运营、统一品牌）/ 私有化（客户环境、部署级品牌）；身份=Keycloak Organizations 单 realm（org=租户）、数据=schema/db-per-tenant、计算=namespace-per-tenant，由 control-plane 编排开通；品牌为部署级。
- **🔴 信息红线（强制）**：本仓内容按公开仓红线管理——**禁止任何甲方/客户可识别信息与项目机密**（真实单位/人员/编号/代号/地点/拓扑/商务条款）；示例数据一律虚构脱敏（`example.com` / `acme` / `tenant-demo`）。判定：任意文件公开到互联网不泄露客户身份或项目机密。

## C. 我们现状与资源

- **研发能力定位**：自研能力不及华为 → **尽量用开源实现、接受二次封装成本**；但**高效选择"前期就值得投入自研"的部分**。三档实现路线：纯开源集成 / 开源封装(半自研) / 自研封装。
- **团队能力 = 大数据开发经验** → **PRD 写到能力级，不写交互级、不规范字段级**；讲清楚"实现要求 + 建议开源组件的使用/组合模式 + 自研封装的精准边界"即可。
- **五大分系统 + 横切/底座**（PRD 各模块的实现归属）：

  | 仓 | 承载 | 关键选型 |
  |--|--|--|
  | data-foundation | 采集/湖仓/计算/Connector SPI | Flink-CDC+Kafka / SeaTunnel+Spark / Paimon+MinIO / Doris |
  | governance | 元数据/元模型/标准/质量/血缘/数据架构 | **自研元模型引擎**(Atlas TypeDef 蓝本)+PG(JSONB)+ES |
  | security | 分类分级/标签/脱敏/审批/审计 | 规则引擎 + Flowable |
  | privacy | MPC/PSI/匿踪 | 隐语 SecretFlow（Python） |
  | control-plane | 多租户开通/配额/生命周期 | Helm SDK+K8s client+Keycloak Admin+ESO |
  | gateway | 南北向网关/OIDC/限流/审计 | APISIX(声明式)+Keycloak |
  | platform-common | 调度/工作流 | DolphinScheduler / Flowable |
  | webui | 唯一前端（双 app） | React19+Vite+AntD v6+AntV |

- **自研 7 概念**（标 `（自研）`，索引见 `03-自研概念总账.md`）：SC-1 元模型引擎（地基）· SC-2 数据资产 · SC-3 数据服务资产化 · SC-4 落标质检前置网关 · SC-5 指标装配引擎 · SC-6 统一数据权限策略 · SC-7 工作空间与租户开通。
- **双前端平面**（详见 `02-双平面职责划分.md`）：**使用平面 console**（租户内用户、org 作用域、租户内生产消费）/ **管理平面 admin**（跨租户运营、不绑 org、独立域名、开通+全局治理）。切分三问：跨租户? / 定规则 vs 用规则? / 爆炸半径平台级?。
- **完整技术选型菜单**：主仓 `docs/architecture/03-技术选型.md`（首选+备选+信创备注）。

## D. PRD 结构与命名约定

```
docs/
├── README.md                 总纲（方法论·导航·写作规范）
├── 00-平台总览/  01 对标矩阵 · 02 双平面 · 03 自研总账 · 04 一期范围
├── 01-管理中心/  02-数据集成/  03-数据架构/  04-数据开发/  05-数据质量/
├── 06-数据目录/  07-数据安全/  08-数据服务/  09-隐私计算/  99-附录/
```

- **模块编号沿用华为 DataArts 呈现顺序**（管理中心→集成→架构→开发→质量→目录→安全→服务），便于对标核对；隐私计算为特色补充。
- **自研文档文件名后缀含 `（自研）`**（可带概念名，如 `（落标网关·自研）`、`（指标装配·自研）`，更利于辨识）；标题/表格标注实现路线 `【纯开源集成】/【开源封装】/【自研封装】`。
- **复合标注 `【开源封装 + 自研前端】`**：开源为内核、但前端可视化原生自研（血缘图 G6 / 编排画布 X6 / 检索门户）时用。"自研前端"是工程实现，**不构成需登记总账的自研"概念"**——不进 `03-自研概念总账.md`。
- 每模块一子目录；子目录内用 `NN-子模块名.md` 编号；目录可加 `README.md` 作模块总览（结构见 §E.3）。
- **跨文档链接约定**：模块内 / 到 `00-平台总览` 用相对路径（`../00-平台总览/…`）；引用主仓架构（`docs/architecture/`）从 `prototype/docs/<模块>/` 出发为 `../../../docs/architecture/…`，或用「主仓 `docs/architecture/xx`」说明性引用（子模块独立 clone 时更稳）。

## E. 写作规范

### E.1 模块 PRD 八节骨架
1. 模块定位与华为对标（定位 + 对标哪个组件/子能力 + 裁剪了什么）
2. 能力清单（P0/P1/P2，能力级，不展开字段/交互）
3. 开源实现选型与组合（组件 → 职责 → 二次封装成本）
4. 自研判定 / 自研封装定义（**含自研**：按 E.2 八维度模板；**无自研**：本节保留为「自研判定」，显式声明四特征不全中的否决理由、不登记总账——不要省略本节）
5. 双平面职责拆分（套 §C 切分三问；只做**本模块**细化切分 + **链接回** `02-双平面 §3`，不重述矩阵整行；自研模块的作用域**机制**写在 §E.2 作用域维度、本节只写「该机制在 admin/console 各承载哪段职责」）
6. 多租户与信创注意（隔离边界 + 信创适配点）
7. 能力级验收要点（可验证、对标得上，不绑控件）
8. 一期范围与暂缓（与 `04-一期范围与优先级.md` 一致）

> **八节是最小骨架，非上限**。①**自研子模块**正文以旗舰样板（`06-数据目录/04-数据资产（自研）.md`、`01-元模型引擎（自研）.md`）的**更细结构为准**（实践约 14–16 节）：四特征自检放第 1 节"定位与对标"内、§E.2 八维度散布其中（构成放"构成分解"节、作用域单列、双平面单列）——**节序以样板为准，八维度为必含要素而非固定节号**。②**无自研模块**仍是严格八节，第 4 节即"自研判定"，用**四特征逐条否决表**（更可核验）说明为何不是自研、不登记总账。③**跨自研概念的联动**（如质量分回写 SC-2、DQC 熔断联动 SC-4）写在第 1 节定位 + 第 4 节边界声明 + 第 7 节验收，不另起节、不越界改他人概念。
精确定义（一句话本质 + 最小管理单元）→ **不是什么**（逐条排除最易混淆的相邻概念）→ 基于哪些开源构成（**三档区分：复用实现 / 仅借设计蓝本(blueprint) / 降级为可选来源**——明确是"用其实现"还是"只借其设计"，避免被误读为封装某开源运行时）→ 生命周期（状态机 + 触发 + 责任方；**若概念是"类型系统/引擎"类、管理的是"定义"而非"实例"，则画其"定义对象"的状态机，并显式声明与实例生命周期的区分**）→ 作用域（公共/私有 · 设计期/运行期 · 双模差异）→ 在哪些业务流程生效（上下游写/读）→ 扩展点（SPI/插件/配置）→ 与相邻概念关系（**双向**：本概念引用/派生/约束谁 + **谁依赖本概念**——"地基"类概念尤其要画反向被依赖关系）。
> 标尺：再有经验的大数据工程师，也要靠这八维度才能动手 → **范围精准、边界精准、可扩展性与可理解性兼顾**。
> 作用域 vs 双平面分工：作用域**机制**（公共继承/私有扩展规则）写在本模板"作用域"维度；§E.1 第 5 节双平面只写"该机制在 admin/console 各承载哪段职责"，二者不重述。

### E.3 模块 README（总览）轻量骨架
每个模块子目录的 `README.md` 是该模块**总览**（非单篇子模块 PRD，不套 E.1 八节），统一结构：
1. 模块定位与对标（对标华为哪个组件 + 承载分系统 + 实现路线总基调）
2. 子模块导航表（子文档 → 一句话职责 → 实现路线 / 自研编号）
3. 关键概览（端到端流程 / 两条链路 / 信息架构，按模块取一**或多**）
4. 与相邻模块的边界（上下游、复用关系，防越界）
5. 双平面与一期范围概述（链接回 `00-平台总览/`，一句话概述、不重述）

> 这是**要素清单非固定标题序**：可按模块加专属节——如安全模块的"治理域审批/数据级权限**归口声明**"、管理中心的"**容器 vs 内容**关系"、横切模块（数据服务）的"承载分系统双归属"（设计期 `data-service` + 运行期 `gateway`/APISIX）。README 只做模块级概述与判定结论，**深度判定/定义仍在各子文档**。

## Step 工作流：新增 / 修订一篇模块 PRD

1. **定位与对标**：在 `01-华为DataArts对标矩阵.md` 找对应组件 → 列子能力 → 标 ✅/✂️/🕐。确认承载分系统（§C 表）。
2. **判定实现路线档位**：能用开源直接/薄封装 → 纯开源集成 / 开源封装；否则进 Step 3。
3. **自研判定（四特征自检）**：① 跨开源封装 ② 带业务定义属性 ③ 无专门开源实现 ④ 平台差异化所在 —— **全中**才是自研。是 → 文件名标 `（自研）`、登记 `03-自研概念总账.md`、按 E.2 八维度写。**能薄封装的别滥用自研**（控制自研成本是基本盘）。
4. **撰写**：按 E.1 八节；写到**能力级**，给**开源组件组合模式**，**不写字段/交互级**；自研部分按 E.2。
5. **双平面切分**：套切分三问，明确公共作用域（管理平面）与租户私有（使用平面）的继承/覆盖规则。
6. **红线自检 + 一致性回链**：跑红线 grep（见下）；P0/P1/P2 与 `04-一期范围.md` 对齐、自研概念回链 `03-自研总账.md`、对标取舍与 `01-对标矩阵.md` 不冲突；有冲突**回改总览文档**。
7. **收尾**：汇总改了哪些文件、对标覆盖了哪些子能力、新增/变更了哪些自研概念；**不在汇总/提交信息中带客户/凭据/真实主机信息**。

红线 grep（提交前必跑，命中即停核查）：
```bash
grep -rniE "星网|甲方真实名|招标|合同编号|立项|真实IP|内部代号" prototype/docs/
```
（通用词"甲方/红线"作策略说明可保留；命中**具体可识别信息**才需处理。）

## 反模式（避免）

- ❌ 写到字段级/交互级/控件级 —— 团队有大数据经验，PRD 是能力 + 开源组合 + 自研边界，不是交互稿。
- ❌ 把能用开源薄封装的能力标成"自研" —— 抬高成本、拖累工期。
- ❌ 复刻华为云专属能力（专属库/共享专享版/企业简单模式/应用层加密） —— 一律裁剪或开源等价替换。
- ❌ 自研概念定义缺维度（没说"不是什么"、没画生命周期、没界定作用域） —— 工程师无法动手。
- ❌ 在模块 PRD 里重述对标矩阵/双平面/自研总账 —— 只链接 `00-平台总览/`，避免漂移。
- ❌ 跨租户能力误放使用平面、或租户内能力误放管理平面 —— 用切分三问校验爆炸半径。
- ❌ 任何甲方可识别信息或项目机密入库 —— 红线，一律虚构脱敏。
