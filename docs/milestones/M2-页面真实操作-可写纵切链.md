# M2 · 页面真实操作（三条可写纵切链）

> 范围：跨 6 仓的**第二个开发里程碑**协调总纲。各仓以**同名 Milestone** 承接本文对应小节。
> 定位：在 M1「只读骨架贯通」之上，补上**第一批"可写"纵切链**——让前端页面能**真实操作业务功能**（填表单→真后端→落库→回显），可向上做"能动手"的演示汇报。**接受功能不完整，但架构不可错、骨架不可返工。**
> 红线：本文与各仓 Issue/分支/提交均为公开内容，禁止任何甲方可识别信息与项目机密（见各仓 `CLAUDE.md`）。

---

## 0. 为什么是这三条（M1 → M2 的缺口）

M1 收官后，11 仓地基（OIDC 登录 / 租户头注入 / 网关 / PG / 单 ns 部署）**已真实**，但**九大模块前端 100% 走 msw 本地 mock，没有任何一个表单能真正写到后端**。M2 的任务就是补这条缺口——但不是九条一起上。

**关键洞察：成本不在"模块数"，在"解锁几个后端服务的持久层 + 写链路"。** 前端九模块当前都是只读占位、改造成本接近；真正的成本差在该模块 owner 后端服务有没有"可写持久层 + REST 写端点"。owner 与模块是**多对一**：

| Owner 服务 | 持久层现状 | 负责模块 | 解锁杠杆 |
|---|---|---|---|
| control-plane | ✅ 已有 JPA+Flyway+entity | 组织管理（租户自管理区段） | 已解锁，近乎免费 |
| governance | ⚠️ PG 连通但无业务 entity | **数据目录·数据架构·数据质量·管理中心（4 模块）** | 解锁一次，惠及 4 模块 |
| data-foundation | ⚠️ 无持久层（仅采集/SPI 框架） | 数据集成·数据开发·(数据服务) | 解锁一次，惠及 2-3 模块 |
| security | ⚠️ PG 连通但无 entity | 数据安全（1 模块） | 杠杆低，后置 |
| privacy | ⚠️ 编排层无库 | 隐私计算 | 需先定 iframe+SSO 架构，后置 |

**M2 据此挑三条最小可写链，覆盖三类 owner 形态**（采集型 / 目录型 / 运营型），用最少"服务解锁"立起**可复用的「列表→新建→提交→回显」纵切范式**，后续模块在已解锁服务内 M3 廉价复制。

## 1. 目标与验收

**统一目标**：在 M1 已贯通的单 ns + 租户上下文之上，跑通**三条独立的"可写"纵切链**，每条都是「浏览器 → gateway(OIDC+租户头) → 真实后端 → PG → 页面回显」，且**跨租户隔离**。

**演示验收（Demo Acceptance，可勾选可验证）**：

1. **数据集成·数据源接入**（锚点）：console「数据集成」页点「新建数据源」→ 填 MySQL 连接信息 →「测试连接」**真实连上 demo MySQL**（连不上有真实错误回显）→ 保存（密码 **AES-GCM 加密**落 PG）→ 列表回显**本租户**数据源 → 点开数据源**列出库表** → 选表**预览前 N 行**真实数据。
2. **数据目录·资产登记**：console「数据目录」页**手动登记/编辑一条元数据资产**（`POST` 真写 PG），`/api/meta/search` 由 mock 改为**真实查 PG 返回本租户资产**。
3. **组织管理·成员管理**：console「组织管理」区段（按角色门控）**邀请/移除一名成员**，经 control-plane **真实写 Keycloak Org membership**，列表回显真实成员。
4. **多租户隔离自检**：以两个租户（acme / 第二个 demo 租户）分别登录，三条链所产生的数据源 / 资产 / 成员**互不可见**（`X-Tenant-Id` 强制过滤）。
5. 三条链各落一个**集成自检**（脚本或 e2e），其余 M1 既有链路与服务**保持 readiness 绿**、经网关可达不回归。

**非目标（明确后置，防镀金）**：
- 数据开发 / 数据质量 / 数据架构 / 管理中心（governance 解锁后 **M3 廉价复制**，本里程碑不做前端）。
- 数据服务（owner 未定，先做归属决策再排期）。
- 数据安全（security 解锁仅惠及 1 模块，性价比低，后置）。
- 隐私计算（需先定 iframe + SSO 嵌入架构，后置）。
- 数据源：增量/实时同步、调度、连接池治理、DM/KingBase/OceanBase 等**非 MySQL 方言的真实实现**、写回与血缘自动采集。
- 凭据治理升级到 ESO/Vault（M2 用应用层加密 + `secret_ref` 间接层**预留**，不做真 ESO）。
- 元数据：自动采集/爬取、血缘自动解析、typedef 驱动的完整元模型引擎（governance #12 后置）。
- 完整 SC-6 行/列裁决、列级权限。

## 2. 已确认的关键决策（不可返工）

> M1 的 D1–D6 继续生效（单 KC User + 多 Org Membership / 切租户=换 token、`X-Tenant-Id` 唯一 / 白标部署级 / emerald / 子仓交付 image 主仓 owns charts / 身份后端真实 OIDC）。M2 新增 D7–D9。

| # | 决策 | 结论 | 影响面 |
|---|---|---|---|
| **D7** | 数据源凭据存储 | **应用层对称加密**（AES-GCM）后存 PG；**主密钥部署期注入**（env/K8s Secret，**绝不入库**）；表结构存 `secret_cipher` + 预留 **`secret_ref` 间接层**，后续可平滑切 ESO/Vault **不改表**。明文密码**绝不落库、绝不进日志/响应**。 | data-foundation；deploy（密钥注入）；红线 |
| **D8** | 连接器中立 | MySQL 是**第一个** connector-SPI 方言实现，**不写死 MySQL**；数据源领域模型与端点**后端中立**（type 驱动方言），新增方言只加插件、不改端点/模型。 | data-foundation（守 AD-18 不返工不变量） |
| **D9** | 写链多租户隔离 | 数据源 / 元数据资产 / 成员**一律按 `X-Tenant-Id` 强制隔离**：写时落 `tenant_id`、读时强制过滤（D2 不变量）。**组织管理 = `apps/console` 内按角色门控的"租户自管理"区段**（非 admin 跨租户运营）；成员落 **Keycloak Org**，经 control-plane Admin API 读写，control-plane 的 PG 不另存成员凭据。 | data-foundation / governance / control-plane / webui |

**必须同步修订项**（否则代码与契约打架）：
- `contracts/` **新增** `openapi/data-foundation-datasources`（producer=data-foundation）+ 对应 ICD；webui SDK 由此生成、gateway 路由据此配。
- `contracts/icd/governance-metadata`（或 openapi）**扩写写入面**（登记/编辑资产），现仅描述 `search` 取数。
- control-plane 契约**补成员管理面**（org membership 邀请/移除），明确"成员 SoT 在 Keycloak、control-plane 仅编排"。

## 5. 跨仓依赖与推进顺序

```
① 契约先行            主仓 contracts 落 datasources/metadata-write/members 三套契约 + registry
        ▼
② 锚点链立范式        data-foundation(JPA+AES+5端点) → gateway 路由 → deploy demo MySQL
                      → webui 数据集成表单接真实 API（先 curl 通，再前端）
        ▼
③ 复制到目录/组织     governance(资产写+真查) / control-plane(成员) 各立写端点 → gateway 路由
                      → webui 复用 CRUD 范式接两模块
        ▼
④ 隔离与收口          两租户隔离自检 + 三条链集成自检 + M1 冒烟不回归
```

**关键依赖**：webui 三模块依赖各自后端端点 + gateway 路由 + 契约；data-foundation 测试连接/预览依赖 deploy 的 demo MySQL；三条链都依赖 contracts 先行。

## 6. 风险与待澄清

- **R1 D7 密钥管理**：M2 用 env 注入主密钥 + 应用层 AES。须确保**明文密码不进日志/响应/异常栈**；主密钥轮换为后置（`secret_ref` 间接层已为此预留）。
- **R2 governance 首个 entity**：governance 此前仅"打通 PG 不建表"，M2 是它**第一次真正落业务表**，schema 要按可扩展（JSONB）设计，避免 #12 元模型引擎落地时返工。
- **R3 成员 SoT 边界**：成员真相在 Keycloak Org，control-plane 仅编排——**严禁** control-plane 另建成员凭据实体（破坏 D1 单 KC User 不变量）。
- **R4 demo MySQL 红线**：种子数据**一律虚构脱敏**（`acme` / `example.com` / 通用业务名），严禁真实样本。
- **待澄清（不阻塞骨架）**：数据服务模块 owner 归属、隐私计算 iframe 嵌入架构、security 业务表设计时机——M3 前澄清，数据模型已按可扩展预留。
